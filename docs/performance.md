# ⚡ Оптимизация производительности Ollama Stack

В этом документе описаны практические методы ускорения генерации, управления ресурсами VRAM/RAM, выбора квантования и настройки Windows/WSL2 для максимальной отдачи от локального ИИ-стека.

---

## 🎯 Что влияет на скорость генерации

| Фактор | Влияние | Как контролировать |
|--------|---------|-------------------|
| **VRAM GPU** | Критическое | Определяет, сколько слоёв модели поместится в видеопамять |
| **Квантование** | Высокое | `q4_k_m` → баланс, `q8_0` → качество, `fp16` → максимальная точность |
| **Context Length** | Среднее | Чем больше `num_ctx`, тем больше RAM/VRAM потребляет промпт+ответ |
| **CPU Threads** | Низкое/Среднее | Влияет только если модель частично выгружена в оперативную память |
| **Диск** | Низкое (только холодный старт) | NVMe ускоряет загрузку модели с 5 до 1–2 сек |

## 🎮 GPU vs CPU: Как работает offloading

Ollama автоматически распределяет слои модели между VRAM и системной RAM.

### 🔹 Полная загрузка в VRAM (рекомендуется)
```json
// %USERPROFILE%\.ollama\config.json
{
  "num_gpu": 99,
  "low_vram": false
}
```
- ✅ Скорость: **30–90 т/с** (зависит от GPU и квантования)
- ✅ Задержка: <100 мс на первый токен
- 📉 Ограничение: VRAM ≥ размера модели + контекста

### 🔹 Частичная выгрузка (VRAM не хватает)
```json
{
  "num_gpu": 40,  // Количество слоёв, помещающихся в VRAM
  "num_thread": 12 // Потоки CPU для оставшихся слоёв
}
```
- ⚠️ Скорость падает в **3–8 раз**
- 📉 Растёт нагрузка на `vmmem` (WSL2/Docker)
- 💡 Используйте `q4_k_m` или `q3_K_M` для экономии VRAM

### 🔹 Режим `low_vram` (для 6 ГБ и менее)
```json
{ "low_vram": true }
```
- Включает дополнительные оптимизации CUDA
- Снижает пиковое потребление VRAM на 15–20%
- Небольшое падение скорости (~10%)

## 📦 Квантование: выбор формата

| Формат | Потеря качества | Экономия памяти | Скорость | Рекомендация |
|--------|----------------|-----------------|----------|--------------|
| `q4_k_m` | <1% | 50–60% | 🔥 Максимальная | ✅ По умолчанию для большинства задач |
| `q5_k_m` | ~0.5% | 40–45% | ⚡ Высокая | Для точных ответов/кода |
| `q8_0` | ~0.1% | 20–25% | 🐢 Средняя | Когда важна максимальная точность |
| `fp16` | 0% | 0% | 🐌 Низкая | Только для отладки/бенчмарков |
| `q2_K` / `q3_K_S` | 2–5% | 70–75% | 🚀 Очень высокая | Для слабых GPU или быстрого черновика |

> 💡 **Правило:** Начинайте с `q4_k_m`. Если не хватает VRAM → `q3_K_M`. Если нужна точность → `q5_k_m`.

## ⚙️ Тонкая настройка конфигурации

### 🔹 `config.json` (Ollama)
Расположение: `%USERPROFILE%\.ollama\config.json`

```json
{
  "num_gpu": 99,
  "num_thread": 12,
  "main_gpu": 0,
  "low_vram": false,
  "num_batch": 512,
  "num_ctx": 8192
}
```

| Параметр | Описание | Рекомендация |
|----------|----------|--------------|
| `num_gpu` | Слои в VRAM | `99` (все) или `0` (CPU) |
| `num_thread` | Потоки CPU | Чётное число, не больше P-cores |
| `main_gpu` | Индекс GPU | `0` (основная карта) |
| `num_batch` | Пакетная обработка | `512` (баланс), `256` (экономия VRAM) |
| `num_ctx` | Длина контекста | `4096` (экономно), `8192` (стандарт), `16384` (тяжёлые задачи) |

### 🔹 Переменные среды Ollama
```powershell
# Разрешить параллельные запросы (по умолчанию 1)
[Environment]::SetEnvironmentVariable("OLLAMA_NUM_PARALLEL", "2", "User")

# Ограничить максимальную VRAM (в байтах, например 6 ГБ)
[Environment]::SetEnvironmentVariable("OLLAMA_MAX_VRAM", "6442450944", "User")

# Применить изменения
Restart-Service -Name Ollama
```

### 🔹 Open WebUI (`docker-compose.yml`)
```yaml
environment:
  - DEFAULT_CONTEXT_LENGTH=8192   # Контекст по умолчанию
  - MAX_TOKENS=4096               # Лимит генерации
  - DISABLE_ANALYTICS=true        # Экономия ресурсов
  - WEBUI_AUTH=True               # Включить аутентификацию
```

## 📊 Мониторинг и бенчмарки

### 🔹 Быстрая диагностика в реальном времени
```powershell
# GPU утилизация и память (обновление 1 сек)
nvidia-smi -l 1

# Какие модели загружены и сколько VRAM используют
ollama ps

# Ресурсы контейнеров и WSL2
docker stats
Get-Process vmmem | Select-Object CPU, WorkingSet64
```

### 🔹 Простой бенчмарк генерации
```powershell
# Замер времени и токенов/сек
$uri = "http://127.0.0.1:11434/api/generate"
$body = @{
  model = "qwen2.5:7b-instruct-q4_k_m"
  prompt = "Напиши краткое объяснение квантового запутывания простыми словами."
  stream = $false
  options = @{ num_ctx = 4096; temperature = 0.3 }
} | ConvertTo-Json

$stopwatch = [System.Diagnostics.Stopwatch]::StartNew()
$response = Invoke-RestMethod -Uri $uri -Method Post -Body $body -ContentType "application/json"
$stopwatch.Stop()

$t_per_sec = [math]::Round($response.eval_count / ($stopwatch.ElapsedMilliseconds / 1000), 1)
Write-Host "⏱️ Время: $($stopwatch.ElapsedMilliseconds) мс | 🚀 Скорость: $t_per_sec т/с | 📦 Токенов: $($response.eval_count)"
```

## 🛠️ Оптимизация Windows и WSL2

| Компонент | Настройка | Эффект |
|-----------|-----------|--------|
| **WSL2 RAM** | `%USERPROFILE%\.wslconfig` → `[wsl2]\nmemory=8GB` | Предотвращает исчерпание RAM хоста |
| **Docker Desktop** | Settings → Resources → WSL 2 backend ✅ | На 20–40% быстрее, чем Hyper-V |
| **План электропитания** | `powercfg -s 8c5e7fda-e8bf-4a96-9a85-a6e23a8c635c` (Высокая производительность) | Снижает троттлинг CPU/GPU |
| **Фоновые приложения** | Отключить аппаратное ускорение в браузере, закрыть тяжёлые IDE | Освобождает 1–2 ГБ VRAM |
| **SSD** | NVMe PCIe 3.0+ для `.ollama\models` | Холодный старт 1–2 сек вместо 5–10 |

Пример `.wslconfig`:
```ini
[wsl2]
memory=8GB
processors=6
swap=2GB
localhostForwarding=true
```

## 📉 Типичные узкие места и решения

| Симптом | Причина | Решение |
|---------|---------|---------|
| Скорость <10 т/с | Модель в RAM, а не в VRAM | Установите `num_gpu = 99`, проверьте `ollama ps` → `fully offloaded` |
| Высокое потребление RAM (>16 ГБ) | Большой `num_ctx` или много параллельных сессий | Уменьшите `num_ctx` до 4096, настройте `OLLAMA_NUM_PARALLEL=1` |
| Зависание при первом запросе | Модель грузится с диска + компиляция CUDA | Нормально для холодного старта. Оставьте ПК включённым на 5 мин для прогрева |
| `vmmem` грузит CPU 100% | WSL2 исчерпал память, идёт свопинг | Настройте `.wslconfig`, закройте лишние контейнеры, увеличьте swap |
| Дроп скорости после 10 мин работы | Thermal throttling GPU | Очистите радиаторы, включите кастомный fan curve, снизьте `num_batch` до 256 |
