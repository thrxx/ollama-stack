# 🏗️ Архитектура Ollama Stack

В этом документе описана внутренняя структура проекта, взаимодействие компонентов и потоки данных.

---

## 📐 Общая схема: 3 уровня

Решение построено по гибридной схеме: **бэкенд (Ollama)** работает нативно в Windows для максимального доступа к железу (VRAM), а **интерфейс (Open WebUI)** работает в контейнере Docker (WSL2) для изоляции и удобства управления.

```
┌─────────────────────────────────────────────────────────┐
│ 3. КЛИЕНТСКИЙ СЛОЙ                                      │
│ • Браузер (Chrome/Edge/Firefox)                         │
│ • http://localhost:3000                                 │
────────────┬────────────────────────────────────────────┘
             │ HTTP/REST API (JSON)
             ▼
┌─────────────────────────────────────────────────────────┐
│ 2. СЛОЙ ИНТЕРФЕЙСА (Docker on WSL2)                     │
│ • Контейнер: ghcr.io/open-webui/open-webui:main         │
│ • Хранение чатов: ./webui-data (Bind Mount)             │
│ • Связь с бэкендом: http://host.docker.internal:11434   │
└────────────┬────────────────────────────────────────────┘
             │ HTTP API (OLLAMA_BASE_URL)
             ▼
┌─────────────────────────────────────────────────────────┐
│ 1. СЛОЙ ИСПОЛНЕНИЯ (Windows Native)                     │
│ • Служба Windows: Ollama (Service)                      │
│ • Модели: %USERPROFILE%\.ollama\models                  │
│ • Конфиг: %USERPROFILE%\.ollama\config.json             │
│ • GPU Access: Direct (CUDA)                             │
└─────────────────────────────────────────────────────────┘
```

## 🧩 Компоненты

### 1. Ollama (Бэкенд)
Работает как **Windows Service**. Запущен из-под текущего пользователя, но в режиме службы.
- **Назначение:** Загрузка моделей, управление VRAM, генерация токенов.
- **Порт:** `11434` (слушает на localhost).
- **Хранение:** Все бинарные файлы моделей лежат в `%USERPROFILE%\.ollama`.
- **Конфигурация:** Управляется через `config.json` (параметры потоков, VRAM).

### 2. Open WebUI (Фронтенд)
Работает внутри **Docker-контейнера**.
- **Назначение:** Чат-интерфейс, управление промптами, история диалогов.
- **Окружение:**
  - Использует переменную `OLLAMA_BASE_URL=http://host.docker.internal:11434` для связи с Windows-хостом.
  - Данные пользователей сохраняются в папку `./webui-data`, которая проброшена в контейнер через `bind mount`.

### 3. Docker Desktop (Транспорт)
Обеспечивает виртуализацию для контейнера WebUI.
- **Backend:** WSL2 (Ubuntu).
- **Сеть:** Использует встроенный DNS `host.docker.internal` для резолвинга IP-адреса хоста Windows изнутри контейнера.

---

## 🔄 Жизненный цикл запроса

1. **Ввод:** Пользователь вводит текст в браузере.
2. **Маршрутизация:** Браузер отправляет POST-запрос на `http://localhost:3000/api/chat`.
3. **Прокси:** Контейнер Open WebUI принимает запрос и формирует промпт.
4. **Туннель:** WebUI отправляет запрос через сеть Docker к `host.docker.internal:11434`.
5. **Инференс:** Служба Ollama получает запрос, подгружает нужные слои модели в VRAM и генерирует ответ с помощью CUDA.
6. **Стриминг:** Ollama возвращает токены потоком (SSE). WebUI рендерит их по мере поступления.

##  Структура данных на диске

Проект разделяет данные приложения и данные нейросети:

| Путь | Тип | Описание |
|------|-----|----------|
| `ollama-stack/docker-compose.yml` | Файл | Конфигурация запуска WebUI |
| `ollama-stack/webui-data/` | Директория (Bind Mount) | SQLite база, чаты, загруженные документы |
| `%USERPROFILE%\.ollama\models\` | Директория (Windows) | Тяжелые файлы моделей (GGUF) |
| `%USERPROFILE%\.ollama\config.json` | Файл | Настройки потоков CPU и VRAM |

---

## ️ Границы изоляции

- **WebUI:** Полностью изолирован внутри контейнера. Не имеет прямого доступа к файловой системе Windows, кроме проброшенной папки `webui-data`.
- **Ollama:** Имеет полный доступ к системным ресурсам Windows (CPU/RAM/GPU) как нативный процесс.
